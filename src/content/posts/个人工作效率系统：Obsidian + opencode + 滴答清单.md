## 概述

我的工作围绕高校信息化建设方案编写与项目管理，这套系统以 **Obsidian** 为中控，串联 **opencode**（AI 写作/分析）和 **滴答清单**（任务收集与跟踪），核心不是写代码，而是**管方案、管任务、管知识**。

```
Obsidian（知识库 + 任务看板）
    ↕  dida-sync 插件双向同步
滴答清单（微信/手机/PC 多渠道收集）
    ↕  MCP 协议
opencode（AI 辅助写方案、分析需求）
```

## 二、任务管理体系

任务管理走"双轨制"：**Obsidian Tasks** + **滴答清单**，通过 `Dida Sync` 插件双向同步。

### 任务收集渠道

```
微信 → 滴答清单公众号（语音/文字创建任务）
手机 → 滴答清单桌面小工具
电脑 → 滴答清单 PC 端 / Obsidian
```

### 任务看板

**`任务看板.md`** 是每日任务驾驶舱，用 Tasks 插件查询渲染：

- **今日待办** — 到期日 = 今天的任务
- **本周待办** — 本周内到期的任务
- **快速添加** — 随时记录新任务

**`任务.md`** 按 PARA 分类展示：

```
📋 值得关注 — 高优/进行中/七天内到期
📋 项目（1. 项目）— 各项目任务
📋 领域（2. 领域）— 持续积累领域
📋 资源（3. 资源）— 参考资料
📋 周期笔记（0. 周期笔记）— 周期性回顾任务
```

### 滴答清单同步

通过 `obsidian-dida-sync` 插件，Obsidian 中的任务和滴答清单双向同步。手机上用滴答清单添加任务 → 自动同步到 Obsidian，反之亦然。

模板示例（`其他/1、模板/dida模板.md`）：

```yaml
---
dida:
  projectId: 5b6u5L-h6YeH6ZuG
  status: all
---
```

## 三、opencode 配置

### 配置文件

`~/.config/opencode/opencode.jsonc`：

```jsonc
{
  "$schema": "https://opencode.ai/config.json",
  "mcp": {
    "ticktick": {
      "type": "local",
      "command": ["npx", "-y", "@ticktick/mcp-server"],
      "environment": {
        "API_TOKEN": "your_token",
        "API_DOMAIN": "api.dida365.com"
      }
    }
  }
}
```

通过 MCP 接入滴答清单后，可以在 opencode 中自然语言操作任务：

> "我今天有哪些任务？"
> "帮我建一个任务：下周五前完成 XX 学校的方案初稿"

### 智能体（Agent）

存放在 `-1、Agents/` 目录，`AGENTS.md` 定义了工作区的上下文和 opencode 行为指令。配合已安装的技能使用：

| 技能 | 用途 |
|------|------|
| **proposal-writer** | 写建设方案、可行性报告 |
| **solution-architect** | 技术架构设计 |
| **project-plan** | 项目计划、WBS |
| **doc-generator** | 会议纪要、验收文档、PPT 大纲 |
| **tech-review** | 方案评审 |
| **humanizer-zh** | 方案去 AI 味，更像人写的 |

## 四、方案编写流程（完整步骤拆解）

这是我的核心工作流，以"写一份高校信息化建设方案"为例，完整走一遍从接到需求到输出 Word 的全程。每个环节拆开讲清楚做什么、用什么、为什么。

---

### 第一步：需求收集与任务创建
所有方案的起点都是同一个动作：**收到需求**。
需求来源可能是客户微信消息、领导口头安排、会议纪要，也可能是招标文件或邮件。不管来源是什么，第一件事不是打开 Obsidian 开始写，而是**先把需求变成一条任务**。
手机微信上，长按消息转发给滴答清单公众号，语音或文字自动创建一条任务。如果是会议中口头安排，直接打开滴答清单手机小组件语音录入："写 XX 学校的智慧课程平台建设方案，周五前初稿"。这条任务会自动同步到 Obsidian 的任务看板。
任务创建后，我在 Obsidian 的 `任务看板.md` 中看到它出现在"本周待办"区。

```markdown
## 方案启动
**项目名称**：{{title}}
**客户**：{{customer}}
**产品线**：{{product}}
**核心需求摘要**：

（从客户沟通记录中粘贴关键需求点）
```

这个模板的好处是强制我在动笔之前先想清楚：客户是谁？用什么产品？什么时候交？避免写到一半才发现方向偏了。
### 第二步：输入需求/开始
资料准备到位后，进入方案编写环节。我直接在 Obsidian 中调出 opencode 面板，开始对话。
**第 1 层：明确任务**
我不会一上来就让 AI 写完整方案，而是先下一条"调研+大纲"指令：

```
@proposal-writer 结合我提供的资料，帮我梳理 XX 学校智慧课程平台的建设需求，
输出一份方案大纲，包含：项目背景、建设目标、建设内容、预期成效、预算估算
五个部分。大纲要贴合教育数字化转型的政策背景。
```

这条指令加载了 `proposal-writer` skill。这个 skill 里预置了建设方案的常见结构、政策文件数据库、方案写作风格指南。AI 理解指令后，先输出一份大纲。
### 第三步：查找产品资料与参考案例

 AI 先做一件事：**把方案需要用到的素材喂给自己**。
这一步非常关键。直接让 AI 写方案而不给参考资料，出来的内容往往空洞无物、偏离产品实际能力。所以我有一个固定的"资料搜集"流程：
**1. 定位产品线**
客户需求通常对应一个或多个产品线。比如客户要"智慧课程平台"，AI就进 `0、公司产品/11、智慧课程/` 目录。这个目录下存放着该产品线的全部资料：产品介绍PPT、建设方案文档、报价方案、技术参数表、成功案例等。
**2. 翻阅已有方案**
在公共方案参考文件夹下面和对应客户目录下，先翻之前给这个客户出过的方案。同一客户的文档风格、用词习惯、重点关注领域通常有一致性，参考之前方案可以保持整体风格的连贯。

### 第四步：查找产品资料与参考案例
**第 2 层：逐段编写**
大纲确认没问题后，我逐段要求 AI 展开。比如：
```
继续写"建设内容"部分。分为六个模块：
1. 智慧教学环境搭建
2. 教学内容智慧化组织
3. 教学辅助智慧化
4. 学习支持智慧化
5. 教学研究智慧化
6. 教学评价智慧化

每个模块写 200-300 字，结合智慧课程产品的实际功能来描述。
不要写成技术说明书，要写成"客户价值"的表述方式。
```

逐段写的好处是：每一段我都能即时 review，方向偏了立刻纠正，不会出现 AI 写完整篇后才发现大方向不对的情况。

**第 3 层：调用 solution-architect 做技术方案**

如果方案中涉及技术架构设计（比如系统部署架构、数据流转方案、集成方案等），我会再加载 `solution-architect` skill：

```
@solution-architect 针对上述建设内容的六个模块，
设计系统的技术架构方案。要求：
1. 画出部署架构的文字描述（前端负载均衡、应用服务集群、数据存储层）
2. 说明各模块之间的数据流转关系
3. 标注关键技术的选型理由
```

**第 4 层：补充项目计划和预算**

需要项目计划表或预算表时，调用 `project-plan` skill：

```
@project-plan 根据上述建设内容，输出项目实施计划 WBS，
包含里程碑节点、各阶段工作内容、交付物清单。
```

需要预算估算时，`proposal-writer` skill 中预置了常见的预算模板，直接调用：

```
在方案最后补充预算估算部分，包含：
硬件/软件/服务三类费用，每类列出明细。
```
### 第五步：参考模板格式输出 Word
方案内容在 opencode 中逐段完成后，要进行最后一步：**生成格式化的 Word 文档**。
**模板文件**
我们有一套标准 docx 模板，存放在 `-1、Agents/templetes/建设方案（格式排版例子）.docx`。这个模板包含了标准的封面页、页眉页脚、标题层级样式、表格样式、正文段落样式等。模板是用 Word 手工制作好的，确保排版质量。
**用 python-docx 填充**
我不会在 Word 里手动复制粘贴，因为 60+ 页的方案排版会疯掉。我用 `python-docx` 写一个脚本，读取模板文档，找到模板中的占位符（如 `{{project_name}}`、`{{content_section_1}}` 等），把 AI 生成的内容逐段填入对应的位置。
opencode 本身支持执行 Python 脚本，所以流程在同一个对话中完成：
```
帮我生成一个 python-docx 脚本：
1. 读取模板文件 `建设方案（格式排版例子）.docx`
2. 在指定位置插入各章节内容
3. 保留模板中的所有样式（字体、字号、颜色、缩进、表格样式）
4. 封面页的项目名称、客户名称、日期自动填充
5. 输出最终文件 `XX学校智慧课程平台建设方案.docx`
```
这个脚本会把之前生成的所有内容一次性填入模板，生成一个排版完整的 Word 文档。整个过程不需要手动调整任何格式
### 第六步：去 AI 味（Humanizer）
Word 文档生成后，还有一道关键工序：**去掉 AI 味儿**。
AI 生成的文字有几个典型的特征：句式结构相似（"通过……实现……"）、用词套路化（"助力""赋能""驱动"反复出现）、段落之间过渡生硬。高校客户大多是教授、专家，对文字风格很敏感，一份"AI 味道过重"的方案，初印象会打折扣。
这一步我调用 `humanizer-zh` skill：
```
@humanizer-zh 审阅这份方案的"建设内容"和"预期成效"两个章节，
按照 Wikipedia "Signs of AI writing" 指南 v2.5.1 标准，
检测并修复 AI 模式。

重点检查：
1. 句式多样性——避免连续使用"通过……实现……"结构
2. 用词自然度——把"助力""赋能""驱动"替换为更自然的表达
3. 段落衔接——让段落之间有自然的过渡，而不是并列堆砌
4. 去掉不必要的数字编号和罗列感
5. 每段开头避免重复的"首先/其次/最后"
```

`humanizer-zh` skill 集成了一套 AI 文本检测规则，覆盖 29 种常见的 AI 写作模式。它会逐句扫描文本，标记出"疑似 AI"的段落，然后给出改写建议。我审核后采纳或微调。
这一步之后，方案读起来就像人写的了——不是完美的文学，但至少是一个专业的人写的方案，而不是机器吐出来的说明书。
### 第七步：交付与归档

Word 文档生成并润色完成后，做三件事：
**1. 自审一遍**
我快速通读全文，重点检查：客户名称有没有写错、产品功能描述是否准确、数据是不是最新版本。AI 偶尔会"编"一些看起来很合理但实际不存在的数据，这一步人工审阅不能省。
**2. 归档到客户目录**
最终版本放到对应的客户目录下，比如 `1、重庆电子科技职业大学/1、启明星专业建设系统/建设方案/`。文件名加上日期版本号，如 `20260611_XX学校智慧课程平台建设方案_v1.0.docx`。
**3. 更新任务状态**
回到滴答清单或 Obsidian 任务看板，把对应的任务标记为"已完成"，记录实际完成时间和交付物路径。周回顾时可以快速统计一周完成了多少份方案。
**4. 记录可复用的素材**
如果这份方案中有些段落写得特别好（比如某个功能模块的描述、某个政策背景分析），我会把它摘录到 `0、公司产品/11、智慧课程/方案素材库.md` 中，下次写同类方案时直接复用或参考。

## 七、快速开始

1. **安装 Obsidian**，打开 `~/CX/` 作为仓库
2. **安装插件**：Tasks、Dida Sync、Dataview、Templater、Periodic PARA

6. **配置智能体**：在 `-1、Agents/` 中定义
7. **开始使用**：方案用 proposal-writer，任务用滴答清单，知识存 Obsidian
